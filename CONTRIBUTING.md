# Вклад в развитие проекта

## Соглашение о коммитах (Conventional Commits)

Этот проект использует [Conventional Commits](https://www.conventionalcommits.org/) для автоматического управления версиями с помощью semantic-release.

Формат:
```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### Типы коммитов:

- `feat`: Новая функция (увеличивает минорную версию)
- `fix`: Исправление бага (увеличивает патч-версию)
- `BREAKING CHANGE`: Критические изменения (увеличивает мажорную версию)
- `docs`: Изменения в документации
- `style`: Изменения, не влияющие на логику (пробелы, форматирование)
- `refactor`: Изменения, не исправляющие баги и не добавляющие фич
- `perf`: Улучшения производительности
- `test`: Изменения в тестах
- `chore`: Другие изменения (например, обновление зависимостей)

### Примеры:

- `feat: добавление новой команды roo-sync`
- `fix: исправление ошибки в обнаружении путей`
- `feat!: изменение формата конфигурационного файла (BREAKING CHANGE)`

## Процесс релиза

1. При пуше в ветку `main` запускается GitHub Actions workflow
2. semantic-release анализирует последние коммиты с момента последнего релиза
3. Определяется новый номер версии на основе conventional commits
4. Создается новый релиз на GitHub и публикация в npm
